# OpenClaw 初赛答辩 · 30 条问答预案

> 评审 2 分钟质询。本文件按 5 个类别 × 6 条 = 30 条整理。
> 每条结构：**问题** → **30 秒应答框架**（一两句结论 + 一个数据 / 案例 / 边界）→ **加分点**（如有）

## A · 技术与平台（6 条）

### A1. 为什么不直接用国家反诈中心 App 的接口或数据，而要重做一个？
- 通用产品针对全国通用规则，本系统补充的是「校园语境」——辅导员、奖助、二手、租房、社团、老乡群等只在校园出现的话术。
- 二者是互补关系：通用规则走国家反诈，校园规则走本系统；前者 App 拦截，后者粘贴即查。
- 我们不替代任何官方渠道，紧急情况一律导向 96110 与南大保卫处 81686110。

### A2. 双路融合检测引擎和大模型直接判别相比，有什么必要性？
- 规则路保证可解释性与稳定性，命中关键词时给出明确依据；向量路覆盖长尾语义变体。
- 综合分 = rule_score + 2.5 × vector_sim。BGE-zh 失败回落 TF-IDF，再失败回落仅规则——三级降级保证可用性。
- 9 条标杆输入的冒烟测试 TOP-1 全部命中（含新增虚假校招类），单次 CPU 推理约 180ms。

### A3. OpenClaw 平台具体提供了什么本地实现做不到的能力？
- 三项原生能力：多模态理解（OCR / QR / 转账界面）、工具调用（RAG / URL / 外部反诈接口 / 脱敏）、流程编排（多 Agent 协同 / 案例回流自动化）。
- 这些能力如果自研，工程量大且容易拼凑。把工程复杂度交给平台，团队可以集中在知识库、提示词、案例标注、产品叙事四件事上。

### A4. 多 Agent 编排相对单一大模型有什么优势？
- 主控 Agent 路由 + 三个子 Agent 各管一类，每个 Agent 的提示词、知识库、调用工具都更聚焦，准确率更高。
- 子 Agent 可独立迭代——文本规则更新不影响截图 OCR 流程。
- 案例回流 Workflow 独立运行，与主检测链路解耦，避免阻塞实时响应。

### A5. token 成本如何控制？大规模上线会不会失控？
- 单次检测 token 边际成本约 0.015 元（嵌入 + 向量检索为主，LLM 解读为辅）。
- 控制策略：常见话术走规则 fast-path 不调用 LLM；命中相同 case 缓存结果；只对长尾或低置信度场景调用大模型解读。
- 南大 4.8 万人按 30% 月活计算，年 token 成本约 2,600 元；含云服务和域名的综合年成本控制在 5,000 元以内。

### A6. 模型回答错误（误报 / 漏报）的兜底机制是什么？
- 红黄绿三档而非二元判定：误报最坏只是「中等风险」提示用户二次核实。
- 系统明确只做识别与提示，不替代决策、不拦截短信。最坏情况是用户多核实一次，不会带来直接损失。
- 每次判定都给出案例 ID 与匹配依据，用户可追溯并反馈。

## B · 数据与知识库（6 条）

### B1. 50 条 KB 太少了，准确率怎么保证？
- 决赛阶段扩到 200+ 条，覆盖国家反诈十大类、新型 AI 诈骗、校园特化（含新增虚假校招）。
- 当前 50 条已经过 9 条标杆输入冒烟测试 TOP-1 全部命中验证。
- 关键不是数量，而是每条 case 强制 source 字段引用公开材料，质量优先。

### B2. 真实校园案例从哪来？数据合规吗？
- 冷启动阶段使用国家反诈手册、其他高校公开通报、南大保卫处材料——全部为公开来源。
- 原型阶段用户输入默认不持久化，避免任何数据合规风险。
- 决赛阶段对接学工部与保卫处，走正式授权渠道获取脱敏后的真实校园案例。

### B3. 如果数据被攻击或泄露怎么办？
- 知识库本身不含 PII，所有 case 都是公开材料，泄露也无敏感数据。
- 用户输入即用即弃；若用户授权回流，由独立的脱敏 Agent 自动处理姓名、手机号、银行卡号、住址、学号等字段后再写库。
- 系统部署在 OpenClaw 与中国移动云端，符合数据本地化与等保要求。

### B4. 怎么处理 AI 换脸、共享屏幕这种新型诈骗？
- KB-016 已覆盖 AI 换脸与拟声诈骗，KB 中也覆盖共享屏幕远控、二次诈骗清退群、币圈元宇宙等 2024-2025 高发新型场景。
- 应对建议强调「电话二次核实 + 家人暗号 + 视频做指定动作」三层判别。
- 知识库持续滚动更新，社区贡献 + 学工部回流双源。

### B5. 数据是否会涉及南大师生的真实隐私？
- 当前 KB 中没有任何南大师生真实数据；所有引用案例均为公开材料。
- 决赛阶段对接保卫处的真实案例必须先经脱敏 Workflow，去除所有可识别信息后才入库。
- 公开 KB 部分用 CC BY 4.0 协议发布，仅需保留来源标注即可被其他高校引用。

### B6. 怎么判断一条新案例是不是值得入库？
- 三道门：① 必须有可验证的公开来源 URL 或正式出版物名称；② 必须能抽象出 keywords + patterns；③ 必须覆盖三步行动建议。
- 详见 CONTRIBUTING.md 贡献指南。
- 知识库版本号严格管理，每次变更可追溯。

## C · 商业与价值（6 条）

### C1. 100 倍 ROI 是不是夸大了？
- 量级而非精确数字：分母（成本）按保守口径取 5,000 元/年——含 token、云服务、域名，不含开发与维护人工；分子（挽回损失）按公安部 TOP3 覆盖 + 保守 30% 干预率取 50-100 万。
- 测算口径已在 PPT 中清楚标注，每个数字都可追溯到公开来源。
- 即使把分母再放大 5 倍按真实商业部署算（含人工运维），ROI 仍然在 20 倍以上量级，这是公益项目的合理水位。

### C2. 这个项目能不能商业化？盈利模式是什么？
- 项目核心定位是公益与教育，不追求商业化。
- 长期价值在于「校园安全数据资产」——脱敏后的本校诈骗模式库，可反哺学工部、保卫处的预警工作，是软实力建设。
- CC BY 4.0 + MIT 双协议允许其他高校复用，形成「校际反诈联盟」生态。

### C3. 怎么推广到全国其他高校？
- 知识库分两层：通用反诈层 + 校园层。其他高校接入只需替换「本地紧急通道」与少量校园术语。
- 决赛阶段将代码 + 提示词 + 知识库全量开源，附详细的接入文档。
- 短期目标 1-3 所兄弟高校；中长期形成开源标准。

### C4. 为什么用户会用这个工具，而不是直接报警？
- 报警是事后处置，本工具是事前识别——一条短信粘进去几秒钟出结果，门槛极低。
- 三步建议明确告诉用户「如果已转账请立即拨打 96110」，工具与官方渠道是导流关系而非竞争关系。
- 微信小程序载体决定了无需下载，零摩擦使用。

### C5. 你的目标用户规模有多大？
- 南京大学在校生 47,911 人是初始用户池；首发聚焦新生群体约 8,000 人/年。
- 嵌入南大官方 App 后覆盖率可达 100%。
- 模式跑通后可向"双一流"高校复制——全国约 320 万双一流在校生潜在市场。

### C6. 项目最大的风险是什么？
- 最大风险是真实校园案例数据获取——学工部与保卫处的对接周期可能较长。
- 应对策略：① 冷启动用国家反诈 + 其他高校公开案例； ② 入围决赛后主动对接，凭借开源原型作为信任凭证； ③ 案例不足时，引入「反诈预警要点」等通用规则补足。

## D · 隐私与边界（6 条）

### D1. 凭什么相信你们不会偷偷存用户数据？
- 原型阶段用户输入默认不持久化——这是代码层面的硬约束，开源后可被验证。
- 用户授权才启动回流，授权弹窗明确告知"将经脱敏处理后用于公益反诈案例库"。
- 公开发布的数据隐私与 AI 安全说明书将明确边界，违反将承担法律责任。

### D2. 脱敏 Agent 怎么保证 PII 被彻底去除？
- 脱敏 Agent 识别替换姓名、手机号、银行卡号、住址、学号、邮箱、身份证号等敏感字段，使用正则 + LLM 双路确保覆盖率。
- 每条入库案例需人工审核确认无残留 PII 才发布。
- 用户可在授权弹窗中预览脱敏后版本，并随时撤回授权。

### D3. 系统会不会被坏人反向利用（比如学习诈骗手法）？
- 知识库内容本身就是公开材料的整合，恶意者无需通过本系统也能获取。
- 知识库强调"识别 + 应对"，不提供任何利于实施诈骗的细节。
- 真正的护城河是「红蓝对抗」式的持续更新——攻防螺旋升级是长期机制。

### D4. 如果有人输入正常对话被误判成诈骗，怎么办？
- 系统输出是「提示」而非「裁定」，最高级别也只是建议用户「核实」与「保留证据」。
- 每次判定提供匹配的关键词与案例 ID，用户可清楚看到「为什么被打标」，自行判断。
- 误报反馈通道：用户在卡片上可一键提交「这不是诈骗」，进入人工审核。

### D5. 系统会不会拦截正常短信或电话，影响通讯？
- 明确不做拦截，只做识别与提示。系统不接管短信流、不修改通话记录。
- 这一边界是产品设计原则之一，写进 README 与隐私白皮书。
- 用户主动「粘进去」才会触发检测——所有决策权在用户。

### D6. AI 给出的建议是否构成法律意见？
- 不构成。系统三步建议明确指向官方渠道：96110 反诈劝阻、南大保卫处 81686110、110 报警、12381 涉诈短信预警。
- 我们的建议是「行动指南」而非「法律咨询」，用户在卡片上能看到明确的免责说明。
- 涉及法律纠纷请咨询专业律师，本系统不替代专业意见。

## E · 团队与可行性（6 条）

### E1. 个人参赛，工程量怎么扛？
- 严守「OpenClaw 原生能力不重复开发」原则，把工程复杂度交给平台。
- 团队精力集中在四件事：知识库整理、提示词工程、案例标注、产品叙事——这些都是个人可以高质量完成的工作。
- 当前 v0.2.1 原型已经跑通，CI 已绿，证明个人参赛是可行的。

### E2. 你的专业背景（经济学）和项目（AI Agent）匹配吗？
- 项目核心不是 AI 算法研究，而是「场景理解 + 内容质量」——这恰恰需要对校园生活有深度感知。
- 经济学训练带来的数据敏感性与逻辑严谨性，在本项目的成本测算、价值评估、用户行为分析中是直接资产。
- 工程实现借助 Claude Code 等 AI 编程辅助工具完成，已经验证可行。

### E3. 后续会不会因为个人精力分散而项目停滞？
- 项目已经全量开源（MIT + CC BY 4.0），即使我个人离开，社区可以继续维护。
- 知识库每条带 source 标注，新成员可快速理解并贡献。
- 决赛阶段会主动招募合作伙伴——法学、新闻、计算机方向同学，形成长期团队。

### E4. 项目和你的学业 / 实习冲突吗？
- 本项目立项时间从今年 5 月开始，作为本科课余探索项目；时间规划上与学业平行而非冲突。
- 实习方面，我在招商证券实习的工作偏数据分析，与本项目技能互补而非竞争。
- 时间管理上的承诺：每周至少 15 小时投入本项目至决赛结束。

### E5. 如果没能入围决赛，项目还会继续吗？
- 会继续。项目本身解决的是真实校园痛点，不依赖于赛事结果。
- 即使没入围，开源仓库会持续维护，KB 会扩到 100+ 条，本地 Gradio 版本会发布到南大学生小范围试用。
- 大赛是加速器而非生命线。

### E6. 你为什么选择 OpenClaw 而不是其他 Agent 平台？
- OpenClaw 在多模态理解（OCR / QR）、工具调用（外部 API 接入）、流程编排（多 Agent 协同）三方面提供原生支持，与本项目需求高度吻合。
- OpenClaw 是国内本土平台，符合数据本地化要求；中国移动云端部署符合等保。
- 大赛本身是与 OpenClaw 团队深度协作的机会——绿色通道、技术支持、社区曝光都是赛事附加价值。

## 🆕 数据严谨性追问（新增）

### 「遭遇电诈不等于被骗。你的 47.4% 数据有问题。」
- 完全同意。所以最新版 PPT 第 9 页用的是更严谨口径——**「受害大学生中约 20% 损失超过 5,000 元」**（南都研究院 2021 调研）。这个数据的分母是"已经发生损失的大学生"，不是"全体大学生"，逻辑链是干净的。
- "遭遇电诈尝试"和"实际受骗损失"是两个数量级——前者覆盖每一条来自陌生号码的诱导短信，后者才是真正的资金损失。我们的产品价值在于"在尝试阶段就识别"，所以二者都是相关的，只是我们在 PPT 上用了更保守的口径。

### 「47,911 人这个数是怎么算的？」
- 来自南京大学官网"南大简介"页面：本科 16,008 + 硕士 19,266 + 博士 11,205 + 留学生 1,432 = 47,911 人。
- 如果按全日制在校生口径，是 40,666 人（2023-2024 学年本科教学质量报告）。两个数字都是公开的，差额来自留学生与非全日制硕博。

### 「年综合成本 ¥5,000 是不是漏算了什么？」
- 已计入：① token 成本（约 ¥2,600/年）；② 云服务（轻量级 ECS 约 ¥600-1,200）；③ 微信小程序认证（¥300）；④ 域名+SSL（¥100-300）。
- 未计入：① 开发与维护人工——按学生公益项目自愿投入处理；② 真实商业部署需要的合规、客服、推广人力；③ OpenClaw 平台费用——大赛阶段享受平台支持，商业化阶段需要按公开报价折算。
- 即使把这些都加上（按全职 1 人估算 ¥150,000/年），ROI 仍然在 5-10 倍量级——对公益项目是合理的。

## ⚠ 评委可能的「刁难型」问题应对

### 「我现场输入一条文本试试」
- 优先引导评委用预设按钮（成功率 100%）：「老师可以试试我们提供的 9 个典型场景，每一条都对应一个真实案例库条目。」
- 如果坚持自由输入命中正常：自然展示结果。
- 如果自由输入未命中：「这条文本目前 PPT 演示版的简化引擎未命中，决赛阶段在 OpenClaw 平台上跑的是 200+ 条 RAG 检索，覆盖更广。」

- 「检测逻辑是真实的关键词 + 模拟向量分匹配，源代码在公开仓库 github.com/hjjbh1314/nju-guardian 可验证。token 流转动画是按 OpenClaw 平台真实调度链路设计的，反映 5 步流转过程。」

- 「现阶段本地原型用关键词 + BGE-zh 向量召回，是为了在没有平台访问权限前先验证可行性。这套引擎在 OpenClaw 上的对应实现是 RAG 检索 + 多 Agent 协同，核心是 OpenClaw 平台的工具调用与流程编排能力。」
