# 5 天搭出一个校园反诈 Agent：双路召回怎么救我于"换说法死"

> 适合投：知乎专栏 · CSDN 博客 · 掘金 · 微信公众号
> 字数：~3500 字 · 阅读时间：~10 分钟
> 副标题候选：从 15 类到 50 类、从规则匹配到向量召回，一个本科生周末项目的工程小记

## 缘起：为什么想做这个

我的招商证券实习是在投行部，每天看研报、看公告，和钱打交道。但回到学校，我才发现身边的同学正在被另一种方式"打交道"——

- 室友差点把 800 块"班费"转给假冒的辅导员
- 朋友被骗去做"刷单返利"，最后欠了网贷
- 学姐毕业转让 MacBook 是骗局……当然不是我学姐，是冒充我学姐的骗子

**国家反诈中心 App 当然存在**。但我也在用，体验是：偏通用、偏官方、偏"宣传册"。我同学的需求不是再看一遍宣传册，是把那条可疑消息**粘进去就出结果**。

刚好 2026 年南京大学有一个 OpenClaw 应用创新大赛。我决定做一个原型——叫**南大数智安全官 (NJU Guardian)**。

申报书 5 月 6 日交了。剩下 5 天，我想把它做成一个可以发到 GitHub 让别人也能用的开源项目。这篇文章记录我这 5 天踩的坑、做的取舍和最后留下的设计。

## 第一天：知识库扩到 50 类

### 一开始我犯了一个错

我最早的 v0.1 版本只有 15 类案例，照着国家反诈中心《2023 版宣传手册》整理。15 条作为 demo 够用，但要开源就太单薄。我想到一个"省力"思路：把其他高校保卫处推文的案例也整理进去——反正都是高校嘛，决赛拿到名次再去合作。

写到一半我意识到：

> 一旦发到 GitHub 上，就等于"公开发表"。

"反正都是高校"的逻辑在开源前提下不成立。我必须能给**每一条 case 提供可核验的公开来源标注**，否则就是侵权风险。

于是我把规则订死：

| 来源 | 能用吗 | 怎么用 |
|---|---|---|
| 国家反诈中心 / 公安部 / 12381 | ✅ | 标注来源，可改写 |
| 央视、新华、澎湃公开报道 | ✅ | 引用要点 + URL |
| 高校保卫处**公开**推文 | ⚠️ | 必须改写 + 标注，不能 copy |
| 高校内部 / 未公开材料 | ❌ | 不碰 |
| 论坛 / 贴吧 / 知乎自述 | ❌ | 隐私+真实性都没保障 |

我把 `source` 字段设成**必填**，且加了 schema 校验。50 条 case 全部带公开来源标注，来源短标统一维护在 `docs/sources.md`。

### 案例结构

每条 case 是一个 JSON 对象：

```json
{
  "id": "KB-016",
  "type": "AI换脸拟声",
  "name": "AI 换脸/拟声视频通话诈骗",
  "risk_level": "high",
  "keywords": ["视频通话", "AI换脸", "AI拟声", "深度伪造", ...],
  "patterns": ["(视频|语音).*(借钱|转账|急用|周转)"],
  "script_examples": [
    "微信视频里的'朋友'哭着说手机被偷急需转账",
    ...
  ],
  "steps": ["信息收集...", "AI合成...", "高压实施..."],
  "why_scam": [
    "AI 视频合成已经低成本化，不再'眼见为实'",
    ...
  ],
  "advice": [
    "任何转账请求都通过电话二次核实",
    "与家人约定专属暗号",
    ...
  ],
  "emergency": ["96110", "110"],
  "source": "公安部 2024 新型电信诈骗预警 + 央视新闻 2024 AI 换脸专题"
}
```

50 类的分布：

- 国家反诈十大类（核心）：10
- 新型 2024-2025（AI 换脸 / 共享屏幕 / 币圈 / 二次诈骗 / 假冒反诈 App）：10
- 校园特化（论文代写 / 留学保录 / 付费内推 / 二手 iPhone / 教务系统钓鱼…）：12
- 其他生活/金融场景：18

## 第二天：双路召回怎么救了我

### 规则匹配的天花板

v0.1 的检测就是**关键词 + 正则**：

```python
def score_case(text, case):
    hits_kw = [k for k in case.keywords if k in text]
    hits_pt = [p for p in case.patterns if re.search(p, text)]
    return len(hits_kw) * 1.0 + len(hits_pt) * 1.5
```

对"辅导员让我代收班费转 800"这种**模板化输入**，规则路准确率接近 100%。但:

> 用户不会按你的关键词说话。

我自己测的时候构造了一句：

```
"微信视频里看见同学但是声音怪怪的让我转 2000 块说手机被偷了"
```

这是一条典型的 **AI 换脸诈骗** 描述。但我的关键词里只有 "视频通话"，匹配命中了；但 "AI 换脸"、"声音克隆"、"深度伪造" 这些专业词根本不在用户语境里。

规则路给这条输入的得分是 1.0，被分到 **低风险**。这就坏了——AI 换脸 2024-2025 高发场景，我的引擎居然识别不出。

### 向量召回的契机

向量召回的本质很朴素：

1. 把每条 case 编码成一个向量（"语义指纹"）
2. 把用户输入也编码成向量
3. 算余弦相似度，越接近的越像

跟规则的关键差异是：**它不需要用户用我预设的词**。"声音怪怪"和"语音克隆"在向量空间里是接近的。

我选了 **BAAI/bge-small-zh-v1.5** 作为编码器：

- 中文优化，95MB（不至于让贡献者跑不起来）
- 余弦相似度的输出在 [0, 1] 区间
- 单次编码 CPU < 50ms

### 双路融合的设计

直接替换规则路用向量？我没敢。两个原因：

1. **规则的可解释性**——告诉用户"你这句话命中了'刷单'、'返佣金'两个关键词"，比"语义相似度 0.71"有说服力得多
2. **向量也会误命中**——在 50 类的语料库里，相似度 0.55 以上的"邻居"经常有 3-5 个

最后做了**双路融合**：

```python
@dataclass
class Match:
    case: dict
    score: float           # 综合排序分
    hit_keywords: list
    hit_patterns: list
    vector_sim: float = 0.0

    @property
    def rule_score(self) -> float:
        return len(self.hit_keywords) * 1.0 + len(self.hit_patterns) * 1.5


def detect(text, top_k=3):
    rule_matches = {c["id"]: score_case(text, c) for c in CASES}
    vector_hits = {h.case_id: h.similarity for h in vector_engine.search(text, top_k=10)}

    fused = []
    for case_id, m in rule_matches.items():
        vec = vector_hits.get(case_id, 0.0)
        combined = m.score + 2.5 * vec
        if m.score >= 1.0 or vec >= 0.55:        # 召回门槛：单路强信号即进
            fused.append(Match(..., score=combined, vector_sim=vec))
    return sorted(fused, key=lambda x: -x.score)[:top_k]
```

风险等级也做了相应调整：

```python
if rule_score >= 3.0:                                  # 规则强命中 → 高
    return "高风险"
if rule_score >= 1.5 and vector_sim >= 0.55:           # 双路确认 → 高
    return "高风险 · 规则+语义双路确认"
if rule_score >= 1.5:                                  # 单路规则中等
    return "中等风险"
if vector_sim >= 0.62 or (rule >= 1.0 and vec >= 0.55): # 单路向量强 / 弱双路
    return "中等风险 · 语义疑似"
return "低风险"
```

回头测那条"声音怪怪让我转 2000"：

- rule_score = 1.0
- vector_sim = **0.63**（KB-016 AI 换脸排第一）
- 综合分 = 1.0 + 2.5 × 0.63 = 2.575
- **新判定：中等风险 · 语义疑似（70% 置信度）**

报告里明确写出"📐 规则 1.0 / 🧠 语义 0.63"，让用户知道为什么被打这个标签。

### 工程上的几个坑

**1. 嵌入缓存**

每条 case 编码一次约 30ms × 50 = 1.5s，但每次启动都重新编码就太蠢了。我用 KB 内容的 SHA256 前 16 位作为缓存指纹，存到 `.embeddings_cache/sbert_xxx.npz`。KB 内容变了缓存自动失效重算。

**2. 三档降级**

GitHub 上跑这个项目的人不一定网络通到 HuggingFace。SBERT 加载失败 → 降级 TF-IDF；TF-IDF 也失败（极少数）→ 跳过向量路只用规则。决不让向量路把整个 demo 拖死。

**3. 字符级 N-gram TF-IDF**

降级用的是 `TfidfVectorizer(analyzer="char_wb", ngram_range=(2, 4))`。中文 word-level 切词依赖 jieba，单装一个 jieba 不值得；字符级 N-gram 在反诈这种"短文本 + 关键词"场景下效果其实不错。

**4. dataclass + importlib 的奇怪坑**

写测试时我用 `importlib.util.spec_from_file_location` 动态加载 `nju_guardian.py`，结果 `@dataclass` 装饰器报 `AttributeError: 'NoneType' object has no attribute '__dict__'`。原因是 `dataclass` 内部 `sys.modules.get(cls.__module__)` 返回 None。手动 `sys.modules['nju_guardian'] = mod` 就好了。

## 第三天：可转发卡片是核心传播功能

工具好不好用，看用户愿不愿意分享。

我加了一个「📤 生成可转发卡片」按钮：

```
🔴 反诈检测预警 · 南大数智安全官
─────────────────
原文：辅导员让我代收班费要我先转 800

风险等级：高风险 · 强烈疑似诈骗（置信度 98%）
匹配类型：冒充辅导员/老师/学长诈骗（KB-008）
检测信号：规则 3.5 / 语义 0.71

📌 三步行动建议：
  1. 立即停止操作，不要回复
  2. 通过电话或当面与本人核实身份
  3. 已转账立即拨打 96110 或南大保卫处 81686110

🛡 触发反诈八个凡是：
  · 第 6 条：凡是自称"领导"主动加 QQ/微信...

📞 紧急联系：
  · 南大保卫处 81686110
  · 96110 反诈劝阻 / 12381 涉诈短信
  · 110 报警
─────────────────
via NJU Guardian · OpenClaw 应用创新大赛参赛 Demo
```

为什么这个比检测本身更重要？因为反诈最难的是**让没遭遇过的人警觉**。被骗的人不愿意承认，警觉的人懒得宣传。给一个**复制即可发到班群**的格式，传播阻力降到了最低。

「💾 下载报告」按钮则是另一个场景：用户被骗了想留证据。我直接生成一个 HTML，带完整样式、可打印（CSS `@media print`），用户在浏览器里 ⌘+P 就能存成 PDF。

## 第四天：开源仓库的工程化

### CI 不要装 PyTorch

这个项目依赖 PyTorch + sentence-transformers 加起来 2GB+。要是 CI 每次都装一遍，反诈的 PR 都没法快速合。

我把 CI 拆成两层：

```yaml
- name: Install minimal deps (skip torch / sentence-transformers)
  run: pip install scikit-learn numpy
- name: Run smoke tests
  run: python demo/tests/test_smoke.py
```

`test_smoke.py` 只测：

1. KB schema（50 条、必填字段、来源齐全、id 不重复）
2. 规则路 12 条经典输入命中
3. 八个凡是覆盖度 ≥ 5 条
4. 每条 case 的 emergency 字段含可识别号码

CI 跑完 < 1 分钟。SBERT 的端到端测试放本地。

### 双许可证

- **代码** MIT —— 标准开源协议
- **知识库 + 视觉素材** CC BY 4.0 —— 允许引用，但**必须保留原始公开来源标注**

LICENSE-KB.md 文件单独说明知识库的引用规则，避免别人 fork 后误以为可以去掉 source 字段。

### CONTRIBUTING.md 的"一条不可让步的规则"

我在贡献指南最显眼的位置写了一条加粗的：

> **所有进入仓库的诈骗案例，必须能给出公开来源 URL、正式出版物名称或可核验的公开来源标注。**

并且明确列出"不允许"的来源：论坛 / 贴吧 / 知乎匿名自述、微信群截图、内部材料、实名自述（除非本人公开授权）。

PR 模板里有勾选框强制贡献者确认这一点。

## 第五天：我学到了什么

### 1. 双路召回是廉价的提升

两路总比一路好，前提是融合方式合理。在我这个反诈场景里，规则提供"为什么命中"的可解释性，向量提供"换说法也能接住"的鲁棒性，两者互补几乎没有 trade-off。

### 2. 数据合规是开源的前提

我最早想"先用了再说"，写了几条改写自其他高校公开推文的 case。后来想清楚开源 = 公开发表的逻辑，全部回退重写，每条都标了 URL。这个返工时间花得值——**没有这一步，项目根本不能开源**。

### 3. 用户感受到的功能 ≠ 技术上做了什么

我在 D2 调了几小时双路融合的阈值，写了 100 行向量引擎代码。用户感受到的只是"现在它认得 AI 换脸了"。

但 D3 我加的「📤 转发卡片」是一个 60 行的字符串拼接函数。用户感受到的是"哇这个我可以发到群里！"。

技术深度和用户价值不是同一个维度。两个都要，但不能混淆。

### 4. 5 天能做到什么程度

- ✅ 50 条公开来源案例
- ✅ 双路召回引擎（含降级）
- ✅ Gradio Web UI（6 个 Tab）
- ✅ 转发卡片 + PDF 报告
- ✅ CI + 双许可证 + 贡献指南
- ✅ 30 秒路演脚本

5 天足以从 demo 推到 GitHub。**前提是已经有清晰的产品思路和 50 类的初步知识库**。

如果让我重做一遍，我会把 D1（知识库扩充）+ D2（向量召回）的顺序保留——**先把数据搞对、再上模型**。在没有 50 条 case 的时候上向量召回，等于让模型在 15 个点的稀疏空间里找最近邻，相似度的区分度根本拉不开。

## 没做的部分

留给 OpenClaw 决赛或 v1.0：

- **真实校园场景回流**：希望和南大保卫处合作，把脱敏后的真实预警喂回知识库。这一步必须等比赛有一定知名度后再谈，免得一开始就给保卫处压力
- **小程序部署**：决赛交付微信小程序版本
- **OpenClaw 多 Agent 协作**：当前只是"检测 → 报告"线性流程；OpenClaw 平台可以拆成「输入解析 / 案例匹配 / 风险评级 / 解读生成」四 Agent 协作
- **真实 OCR 与 URL 风险接口**：当前 OCR 用本地 tesseract，URL 用启发式规则；生产版可接入腾讯安全 / 阿里云风险 API
